## 2026-07-24-Q3

**Question:** When should the TUI add a submitted user message to the visible transcript?

## 2026-07-24-A3

**User answer:** Fix the TUI so it does not display an uncommitted user message.

**Decision:** Submitting input sends a command and may update transient activity state, but it does not add a user transcript entry. The TUI adds the user entry only after receiving `TurnStarted`, which the Agent emits after `ConversationContext` successfully appends the message. If append fails, the TUI presents the Agent error without presenting the user message as committed.

**Reason:** `ConversationContext` is the only authority for committed conversation history, and the TUI projects committed state rather than creating it.

**Applied:** `App::submit_input` no longer adds a user timeline entry. `App::apply_agent_event` adds it from the post-commit `TurnStarted` event, with tests covering the boundary.

## 2026-07-24-Q4

**Question:** How should the config path test avoid depending on credentials from the developer's process environment?

## 2026-07-24-A4

**User answer:** Fix the failing test.

**Decision:** Make the test seed a complete `config.toml` inside its temporary application directory, then verify that `Config::load_or_create_in_dir` loads that file and keeps it at the binary-supplied path. Do not mutate process-wide environment variables in the test.

**Reason:** Unit tests must be deterministic and safe to run in parallel. Depending on `OPENAI_API_KEY`, `OPENAI_BASE_URL`, or `OPENAI_MODEL` from the invoking shell makes the test environment-dependent, while changing those global variables inside a parallel test could affect other tests.

**Applied:** The config path test now uses file-backed test credentials and no longer requires external environment variables.
